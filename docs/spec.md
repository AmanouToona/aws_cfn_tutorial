# 仕様書

1. 目標
   このプロジェクトは、AWS 上で運用可能なセキュアなフロントエンドを伴うサーバーレス SaaS 風 アプリケーションを、Infrastructure as Code により構築・デプロイ・運用できる状態を目指す。

2. 最終的なゴール
   CloudFormation / SAM を用いた再現可能なフルスタックインフラの構築

WAF、CloudFront（OAC）、API Gateway を組み合わせた多層防御の実現

GitHub Actions 等を用いたフロントエンド・バックエンドの自動デプロイと CloudFront キャッシュの自動無効化 (Invalidation)

OIDC による安全な AWS 認証

PKCE を導入したセキュアなフロントエンド認証フローの実装

認証付き API とレート制限、テスト自動化、運用監視の基盤整備

3. 最終アーキテクチャ
   3.1 インフラ構成
   Bootstrap スタック

デプロイアーティファクト用バケットおよびフロントエンド用 S3 バケットの作成

WAF スタック

全体の入り口を守る Web ACL の定義

フロントエンド・配信スタック

CloudFront ＋ OAC による S3 ホスティングの保護。WAF との連携

アプリケーションスタック

API Gateway ＋ Lambda ＋ Cognito ＋ DynamoDB ＋ Secrets Manager

3.2 セキュリティ構成
GitHub Actions から AWS へは OIDC で AssumeRole する

フロントエンドの S3 バケットはパブリックアクセスを一切禁止し、CloudFront OAC のみ許可する

すべてのパブリックエンドポイント（フロントエンド・API）は AWS WAF によって保護される

フロントエンドから Cognito への認証は、XSSによるトークン奪取リスクを低減するため、PKCE 付き認可コードフロー を採用する

4. 最終機能要件
   (省略：既存の API 機能、ユーザー・課金・レート制限、データ管理は維持)

4.4 フロントエンド要件 (新規)
アプリケーションへのアクセスはすべて HTTPS (TLS 1.3推奨) で暗号化されていること

ユーザープールと連携し、ログイン状態に応じた画面遷移（未ログイン時はランディングページ、ログイン後はダッシュボード）が行えること

レート制限超過時（429 Too Many Requests）や WAF によるブロック時（403 Forbidden）に、ユーザーに適切なエラー画面を表示できること

5. デプロイ要件
   すべての構成要素を CloudFormation / SAM で管理する

デプロイは以下の順序で自動化されていること

bootstrap (S3 バケット等の作成)

waf (Web ACL の作成)

application (バックエンド API & Cognito の展開)

frontend-dispatch (CloudFront と WAF/S3/API の紐付け)

フロントエンドコードのビルド ＆ S3 アップロード ＆ CloudFront キャッシュ無効化

6. テスト要件
   ユニットテスト・統合テストによる Lambda や API の検証

WAF の動作検証（例：不正なリクエストを模したスクリプトを投げて 403 エラーが返ることを確認する）

フロントエンドの認証・認可の統合テスト

7. 運用要件
   CloudWatch Logs による Lambda/API Gateway のログ監視

AWS WAF のサンプリングログまたは CloudFront アクセスログの有効化と、不正アクセスの可視化

8. 非機能要件
   高セキュリティ: 長期クレデンシャルの排除、エッジでの防御、Origin の隠蔽、フロントエンド認証のセキュア化を徹底する

保守性・再現性・拡張性の維持

9. 完成条件
   フロントエンド画面が CloudFront 経由で正常に表示され、WAF で守られていること

画面から Cognito を通じてセキュアにログインし、認証付き API（/secret, /mixed）を叩いて結果が表示できること

レート制限やセキュリティブロックが設計通りに機能すること

4. 最終機能要件
   4.1 API 機能
   認証不要の公開 API

Cognito 認証が必要な保護 API

認証付き/未認証の両方に対応する統合 API

各エンドポイントにおいて、認証状態に基づいたレート制限を動的に適用する

4.2 ユーザー・課金・レート制限
Cognito ユーザー情報をもとにユーザー識別を行う

認証状態によるアクセス枠の差異化

未ログイン（未認証）ユーザー: レート制限 10 リクエスト（IP アドレス等で識別）

ログイン（認証済み）ユーザー: レート制限 100 リクエスト（Cognito ユーザー ID 単位で識別）

UsersTable により tier 情報を管理し、さらに上位プラン（pro / enterprise 等）に応じたレート制限の拡張を可能とする構造にする

秒単位または月単位の制限を併用し、API Gateway の使用量プラン (Usage Plan) や DynamoDB (RateLimitTable) を活用して状態を管理・評価する

4.3 データ管理
DataRecordsTable に API 実行結果を保存する

RateLimitTable でレート制限状態を記録・更新する（API Gateway ネイティブ機能で不足する場合の補完）

UsersTable でユーザー属性・利用プランを管理する

4.4 フロントエンド要件
アプリケーションへのアクセスはすべて HTTPS (TLS 1.3推奨) で暗号化されていること

ユーザープールと連携し、ログイン状態に応じた画面遷移（未ログイン時はランディングページ、ログイン後はダッシュボード）が行えること

レート制限超過時（429 Too Many Requests）や WAF によるブロック時（403 Forbidden）に、ユーザーに適切なエラー画面を表示できること

5. デプロイ要件
   すべての構成要素を CloudFormation / SAM で管理する

デプロイは以下の順序で自動化されていること

bootstrap (S3 バケット・IAM の作成)

waf (Web ACL の作成)

application (バックエンド API & Cognito & DynamoDB の展開)

frontend-dispatch (CloudFront と WAF/S3/API の紐付け)

フロントエンドコードのビルド ＆ S3 アップロード ＆ CloudFront キャッシュ無効化

6. テスト要件
   ユニットテストで Lambda の主要ロジックを検証する

統合テストで API の挙動を確認する

未認証時（制限 10）と認証時（制限 100）のレート制限超過ケースが、それぞれ正しく 429 エラーを返すことを検証する

WAF の動作検証（不正なリクエストを模したスクリプトを投げて 403 エラーが返ることを確認する）

7. 運用要件
   CloudWatch Logs による Lambda/API Gateway のログ監視とエラー追跡

AWS WAF のサンプリングログまたは CloudFront アクセスログの有効化と、不正アクセスの可視化

デプロイ履歴の管理とセキュリティ設定のコード化

8. 非機能要件
   高セキュリティ: 長期クレデンシャルの排除、エッジでの防御、Origin の隠蔽、フロントエンド認証のセキュア化を徹底する

再現性: 同じテンプレートで同じ構成を再作成できる

保守性と拡張性: スタックを適切に分割し、将来的な課金機能や新プランの追加に耐えうる構造とする

9. 完成条件
   フロントエンド画面が CloudFront 経由で正常に表示され、WAF で守られていること

画面から Cognito を通じてセキュアにログインできること

未ログイン状態で API を連続実行した際、10 回の制限で 429 エラーが発生すること

ログイン状態で API を連続実行した際、100 回まで正常に処理され、それを超えると 429 エラーが発生すること

CI/CD パイプラインが自動でデプロイを完遂できること
